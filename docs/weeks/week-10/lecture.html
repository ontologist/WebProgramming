3. The Problem with Our Current Code / 現在のコードの問題点

Look at the top of your game.htm. The ball is described by several separate global variables that float around in the script:

game.htmの上部を見てください。ボールは、スクリプト内に散らばった いくつかの別々のグローバル変数で表現されています： 

// The ball is scattered across many separate variables
// ボールは多くの別々の変数に散らばっている
var x = 240;          // ball position X / ボールの位置X
var y = 160;          // ball position Y / ボールの位置Y
var dx = 2;           // ball speed X / ボールの速度X
var dy = -2;          // ball speed Y / ボールの速度Y
var radius = 10;      // ball size / ボールのサイズ

function drawBall(x, y, radius) { /* ... */ }

These loose variables cause real problems as a program grows:

これらのバラバラな変数は、プログラムが大きくなると本当の問題を引き起こします：

  • They are not grouped. x, y, dx, dy, and radius all belong to "the ball", but nothing in the code says so.
    グループ化されていない。これらはすべて「ボール」に属しますが、コードはそう言っていません。
  • What if you want a second ball? You would have to copy every variable: x2, y2, dx2, dy2, radius2... messy!
    2つ目のボールが欲しい場合、すべての変数をコピーする必要があります — 大混乱！
  • Behavior is separated from data. The numbers live at the top, but drawBall() and the movement (x += dx) live far away in draw().
    振る舞いがデータから分離されている。数値は上部にあり、描画と移動は離れた場所にあります。
  • Names collide. A short name like x is a global — any other part of the program could accidentally change it.
    名前が衝突する。xのような短い名前はグローバルで、他の部分が誤って変更する可能性があります。

The solution is to bundle the ball's data and behavior together into one unit. That unit is called an object, and the blueprint for making such objects is a class.

解決策は、ボールのデータと振る舞いを1つの単位にまとめることです。 その単位をオブジェクトと呼び、そのオブジェクトを作る設計図をクラスと呼びます。

4. What is Object-Oriented Programming? / オブジェクト指向プログラミングとは？

Object-Oriented Programming (OOP) is a way of organizing code around objects — self-contained units that bundle together:

オブジェクト指向プログラミング（OOP）は、オブジェクトを中心に コードを整理する方法です。オブジェクトは以下をまとめた自己完結型の単位です：

  • Data (called properties) — what the object knows (a ball knows its position, speed, size).
    データ（プロパティと呼ぶ）— オブジェクトが知っていること（ボールは位置、速度、サイズを知っている）。
  • Behavior (called methods) — what the object can do (a ball can draw itself, move, reset).
    振る舞い（メソッドと呼ぶ）— オブジェクトができること（ボールは自分を描画し、動き、リセットできる）。

The Real-World Analogy / 現実世界のアナロジー

Think about a real ball you can hold in your hand:

手に持てる本物のボールを考えてみましょう：

  • Properties (what it knows / data): its color, its size, where it is, how fast it is moving.
    プロパティ（データ）：色、サイズ、位置、移動の速さ。
  • Methods (what it does / behavior): it can roll, bounce, and stop.
    メソッド（振る舞い）：転がる、跳ねる、止まる。

In OOP, we model the game ball the same way: one object that holds its own position and speed, and knows how to draw, move, and reset itself.

OOPでは、ゲームのボールも同じようにモデル化します — 自分の位置と速度を保持し、 自分を描画・移動・リセットする方法を知っている1つのオブジェクトです。

Why OOP Helps / OOPが役立つ理由

Benefit / 利点 What it means / 意味
Organization
整理		 Related data and behavior live together in one place, not scattered. / 関連するデータと振る舞いが1か所にまとまる。
Reusability
再利用性		 Write the blueprint once, create as many balls as you like. / 設計図を一度書けば、いくつでもボールを作れる。
Readability
読みやすさ		 ball.move() reads like a sentence — the code explains itself. / ball.move()は文章のように読める。
Maintainability
保守性		 To fix the ball, you only look in one place: the Ball class. / ボールを直すには、Ballクラスの1か所を見るだけ。

OOP is not entirely new to you. In Session 8 you used object literals for each brick:

OOPは全く新しいものではありません。セッション8では、各ブロックにオブジェクトリテラルを使いました： 

// An object literal: data grouped with { } and key: value pairs

And the canvas and ctx you use every session are objects too — ctx.fillRect(...) calls a method on the ctx object!

毎回使うcanvasとctxもオブジェクトです — ctx.fillRect(...)はctxオブジェクトのメソッドを呼び出しています！

An object literal is great for one object. But when you need many objects that share the same structure and behavior (many balls, many bricks, many paddles), you want a blueprint. That blueprint is a class.

オブジェクトリテラルは1つのオブジェクトには最適です。しかし、同じ構造と振る舞いを共有する 多数のオブジェクトが必要なときは、設計図が欲しくなります。それがクラスです。

6. Class vs Object: Blueprint vs Building / クラス対オブジェクト：設計図対建物

The most important idea in this session:

このセッションで最も重要な考え方：

  • A class is a blueprint (設計図) — a plan that describes what every ball will have and do. A blueprint is not a real ball; you cannot play with a blueprint.
    クラスは設計図です — すべてのボールが持つもの・することを記述した計画。設計図自体は本物のボールではありません。
  • An object (also called an instance) is a real ball built from the blueprint. You can make many real balls from one blueprint.
    オブジェクト（インスタンスとも呼ぶ）は、設計図から作られた本物のボールです。1つの設計図から多数の本物のボールを作れます。
Blueprint (Class) / 設計図（クラス） Building (Object / Instance) / 建物（オブジェクト）
A house plan on paper An actual house you can live in (you can build many)
A cookie cutter Each cookie it stamps out
class Ball { ... } let ball = new Ball(...)

7. Writing a Class / クラスの書き方

The basic structure of a class looks like this:

クラスの基本構造は次のようになります：

class Ball {
    constructor() {
        // set up the object's starting properties here
        // ここでオブジェクトの初期プロパティを設定
    }

    // methods go here (the things a ball can do)
    // メソッドはここに（ボールができること）
    draw() {
        // ...
    }
}

Key Rules / 重要なルール

  • Use the keyword class, followed by a name. By convention, class names start with a Capital letter (Ball, not ball).
    classキーワードの後に名前を書きます。慣習として、クラス名は大文字で始めます。
  • Everything for the class goes inside the curly braces { }.
    クラスのすべては中括弧{ }の中に入れます。
  • Methods are written like functions, but without the function keyword, and there are no commas between them.
    メソッドは関数のように書きますが、functionキーワードなしで、メソッド間にカンマは不要です。

8. The constructor() / コンストラクター

The constructor() is a special method that runs automatically the moment you create a new object. Its job is to set up the object's starting properties.

constructor()は、新しいオブジェクトを作成した瞬間に自動的に実行される 特別なメソッドです。その役割はオブジェクトの初期プロパティを設定することです。 

class Ball {
    // The constructor receives starting values as parameters
    // コンストラクターは初期値をパラメーターとして受け取る
    constructor(x, y, radius) {
        this.x = x;            // store the starting X / 開始Xを保存
        this.y = y;            // store the starting Y / 開始Yを保存
        this.radius = radius;  // store the size / サイズを保存
        this.dx = 2;           // default speed X / デフォルト速度X
        this.dy = -2;          // default speed Y / デフォルト速度Y
        this.color = "yellow"; // default color / デフォルト色
    }
}

Notice we also remember the starting position so we can reset later. The constructor is the perfect place to give every property a sensible starting value.

後でリセットできるように開始位置を覚えておく点にも注目。 コンストラクターは、すべてのプロパティに適切な初期値を与える絶好の場所です。

9. The this Keyword / thisキーワード

Inside a class, this means "the specific object I am working on right now." It is how an object refers to its own properties.

クラス内で、thisは「今まさに操作しているこの特定のオブジェクト」を意味します。 オブジェクトが自分自身のプロパティを参照する方法です。

Imagine a classroom of students all doing the same worksheet. When a student writes "my name" at the top, "my" means that student — not the whole class. this works the same way: it means this one object, not all objects.

同じワークシートをやっている教室の生徒たちを想像してください。生徒が「私の名前」と書くとき、 「私の」はその生徒を意味します — クラス全体ではありません。 thisも同じで、すべてのオブジェクトではなくこのオブジェクト1つを意味します。

When you have two balls, each one has its own this.x. Calling ballA.move() changes ballA's this.x; calling ballB.move() changes ballB's this.x. They never get confused.

2つのボールがあるとき、それぞれが独自のthis.xを持ちます。ballA.move()は ballAのthis.xを変更し、ballB.move()はballBの this.xを変更します。混同することはありません。

Common mistake / よくある間違い: Forgetting this. inside a class. Writing x = x + dx does not update the object's property — you must write this.x = this.x + this.dx.

クラス内でthis.を忘れること。x = x + dxではオブジェクトのプロパティは 更新されません — this.x = this.x + this.dxと書く必要があります。

10. Methods vs Functions / メソッドと関数

A method is simply a function that belongs to an object. The difference is mostly about where it lives and how you call it.

メソッドとは、単にオブジェクトに属する関数のことです。 違いは主にどこにあるかとどう呼び出すかです。

Function / 関数 Method / メソッド
Lives where? / どこにある？ On its own, anywhere in the script Inside a class / object
How to call? / 呼び方 drawBall(x, y, radius) ball.draw()
Gets its data from? / データの取得元 Parameters passed in this (its own properties)

This is the big shift: instead of drawBall(x, y, radius) where you pass the data in, you write ball.draw() — the ball already knows its own x, y, and radius through this.

これが大きな転換です。データを渡すdrawBall(x, y, radius)の代わりに、 ball.draw()と書きます — ボールはthisを通じて、自分のx、 y、radiusを既に知っているのです。

11. Creating Objects with new / newでオブジェクトを作成

A class is just a blueprint — it does nothing by itself. To build a real object from it, use the new keyword. This runs the constructor and gives you back a fresh object.

クラスは設計図に過ぎず、それ自体は何もしません。そこから本物のオブジェクトを作るには、 newキーワードを使います。これはコンストラクターを実行し、新しいオブジェクトを返します。 

// new runs the constructor and builds an object
// new はコンストラクターを実行してオブジェクトを作る
let ball = new Ball(240, 160, 10);

// Now we can read its properties:
// プロパティを読み取れる：
console.log(ball.x);       // 240
console.log(ball.color);   // "yellow"

// And we can make MORE balls from the same blueprint!
// 同じ設計図からもっとボールを作れる！
let ball2 = new Ball(100, 50, 15);
let ball3 = new Ball(400, 200, 8);

This is the payoff / これが見返り: three lines, three completely independent balls, each with its own position, speed, and size — no copy-pasting x2, x3, etc.

3行で、完全に独立した3つのボール — それぞれ独自の位置、速度、サイズを持ち、 x2、x3などをコピペする必要はありません。

12. Building the Complete Ball Class / 完全なBallクラスの構築

Now let's refactor the ball from your game step by step. We will give it three methods: draw(), move(), and reset().

それでは、ゲームのボールを段階的にリファクタリングしましょう。3つのメソッドを与えます： draw()、move()、reset()。

1 The Constructor / コンストラクター

Group all the ball's data into properties. Remember the start position for reset() later.

ボールのすべてのデータをプロパティにまとめます。後のreset()のために開始位置を覚えておきます。

class Ball {
    constructor(startX, startY, radius) {
        this.x = startX;          // current position X / 現在位置X
        this.y = startY;          // current position Y / 現在位置Y
        this.startX = startX;     // remember start for reset / リセット用に記憶
        this.startY = startY;
        this.radius = radius;     // size / サイズ
        this.dx = 2;              // speed X / 速度X
        this.dy = -2;             // speed Y / 速度Y
        this.color = "yellow";    // fill color / 塗りつぶし色
    }
}

2 The draw() Method / draw()メソッド

This is the old drawBall() function — but now it reads from this instead of taking parameters. It needs the drawing context ctx to draw on the canvas.

これは古いdrawBall()関数ですが、パラメーターを取る代わりにthisから読み取ります。 キャンバスに描画するため、描画コンテキストctxを必要とします。 

    // Draw the ball using its own properties
    // 自身のプロパティを使ってボールを描画
    draw(ctx) {
        ctx.beginPath();
        ctx.arc(this.x, this.y, this.radius, 0, Math.PI * 2);
        ctx.fillStyle = this.color;
        ctx.strokeStyle = "blue";
        ctx.fill();
        ctx.stroke();
        ctx.closePath();
    }

3 The move() Method / move()メソッド

This replaces x += dx; y += dy; and also handles bouncing off the four walls (the same logic from Session 6), all inside the ball itself.

これはx += dx; y += dy;を置き換え、4つの壁での跳ね返り（セッション6と同じロジック）も すべてボール自身の中で処理します。 

    // Move the ball and bounce it off the walls
    // ボールを動かし、壁で跳ね返らせる
    move(canvasWidth, canvasHeight) {
        // Bounce off left and right walls / 左右の壁で跳ね返る
        if (this.x + this.dx + this.radius > canvasWidth ||
            this.x + this.dx - this.radius < 0) {
            this.dx = -this.dx;
        }
        // Bounce off top and bottom walls / 上下の壁で跳ね返る
        if (this.y + this.dy - this.radius < 0 ||
            this.y + this.dy + this.radius > canvasHeight) {
            this.dy = -this.dy;
        }
        // Update the position / 位置を更新
        this.x += this.dx;
        this.y += this.dy;
    }

4 The reset() Method / reset()メソッド

When the player loses a life, send the ball back to its starting position. Because we saved startX and startY in the constructor, this is easy.

プレイヤーがライフを失ったとき、ボールを開始位置に戻します。コンストラクターでstartXと startYを保存したので、これは簡単です。 

    // Send the ball back to its starting position
    // ボールを開始位置に戻す
    reset() {
        this.x = this.startX;
        this.y = this.startY;
        this.dx = 2;
        this.dy = -2;
    }

The Complete Class / 完全なクラス

Here are all the pieces assembled into one clean, reusable Ball class:

すべての部品を1つのきれいで再利用可能なBallクラスに組み立てました：

class Ball {
    constructor(startX, startY, radius) {
        this.x = startX;
        this.y = startY;
        this.startX = startX;
        this.startY = startY;
        this.radius = radius;
        this.dx = 2;
        this.dy = -2;
        this.color = "yellow";
    }

    draw(ctx) {
        ctx.beginPath();
        ctx.arc(this.x, this.y, this.radius, 0, Math.PI * 2);
        ctx.fillStyle = this.color;
        ctx.strokeStyle = "blue";
        ctx.fill();
        ctx.stroke();
        ctx.closePath();
    }

    move(canvasWidth, canvasHeight) {
        if (this.x + this.dx + this.radius > canvasWidth ||
            this.x + this.dx - this.radius < 0) {
            this.dx = -this.dx;
        }
        if (this.y + this.dy - this.radius < 0 ||
            this.y + this.dy + this.radius > canvasHeight) {
            this.dy = -this.dy;
        }
        this.x += this.dx;
        this.y += this.dy;
    }

    reset() {
        this.x = this.startX;
        this.y = this.startY;
        this.dx = 2;
        this.dy = -2;
    }
}

Now look how much simpler and cleaner the main game code becomes. Compare the "before" and "after".

メインのゲームコードが、どれだけシンプルで読みやすくなるか見てください。「変更前」と「変更後」を比べましょう。

Before (Procedural) / 変更前（手続き型）

var y = 160;
var dx = 2;
var dy = -2;

    y += dy;

After (Object-Oriented) / 変更後（オブジェクト指向）

// Create one ball object from the blueprint
// 設計図から1つのボールオブジェクトを作成
let ball = new Ball(240, 160, 10);

function draw() {
    // 1. CLEAR / クリア

    // 2. The ball draws and moves ITSELF
    //    ボールが自分自身を描画して移動する

    ball.move(canvas.width, canvas.height);
}
setInterval(draw, 10);

14. Assignment 8 Details / 課題8の詳細

Assignment 8: Refactor the Ball into a Class / 課題8：ボールをクラスにリファクタリング

Objective / 目的:

Take the procedural ball code from your Breakout game and refactor it into a clean, reusable Ball class — without changing how the game looks or plays.

ブレイクアウトゲームの手続き型のボールコードを、きれいで再利用可能なBallクラスに リファクタリングしましょう — ゲームの見た目や動作は変えずに。

Requirements / 要件:

  1. Define a Ball class with a constructor() that sets x, y, radius, dx, dy, and color as properties on this.
    thisにプロパティを設定するconstructor()を持つBallクラスを定義する。
  2. Write a draw() method that draws the ball using its own properties (no parameters except ctx).
    自身のプロパティを使ってボールを描画するdraw()メソッドを書く。
  3. Write a move() method that updates the position and bounces off all four walls.
    位置を更新し4つの壁で跳ね返るmove()メソッドを書く。
  4. Write a reset() method that returns the ball to its starting position and speed.
    ボールを開始位置と速度に戻すreset()メソッドを書く。
  5. Create the ball with new (let ball = new Ball(...)) and use ball.draw() and ball.move() in your game loop. Remove the old global ball variables.
    newでボールを作成し、ゲームループでball.draw()とball.move()を使う。古いグローバル変数は削除する。

Challenge (optional) / チャレンジ（任意）:

Create two or three balls from the same class, each with a different start position, size, and color, and draw them all in the loop. Notice you did not rewrite any ball logic!

同じクラスから2〜3個のボールを、それぞれ異なる開始位置・サイズ・色で作成し、 ループですべて描画してみましょう。ボールのロジックを一切書き直していないことに気づくはずです！

MDNのクラスガイドとW3SchoolsのJavaScriptクラスチュートリアルを読んでください。

15. Common OOP Mistakes / よくあるOOPのミス

Mistake 1: Forgetting this. / this.を忘れる

// WRONG -- "x" is undefined inside the method! / 間違い -- メソッド内で "x" は未定義！
move() {
    x = x + dx;       // does NOT touch the object's property
}

// CORRECT / 正解
move() {
    this.x = this.x + this.dx;   // updates THIS object's property
}

Mistake 2: Forgetting new / newを忘れる

// WRONG -- without new, you do NOT get an object / 間違い -- new なしではオブジェクトを得られない
let ball = Ball(240, 160, 10);   // error / エラー

// CORRECT / 正解
let ball = new Ball(240, 160, 10);

Mistake 3: Putting commas between methods / メソッド間にカンマを置く

// WRONG -- no commas between methods in a class! / 間違い -- クラス内のメソッド間にカンマは不要！
class Ball {
    draw() { /* ... */ },     // <-- remove this comma!
    move() { /* ... */ }
}

// CORRECT / 正解
class Ball {
    draw() { /* ... */ }      // no comma
    move() { /* ... */ }
}

Mistake 4: Lowercase class name / 小文字のクラス名

// Works, but breaks convention / 動くが慣習に反する
class ball { }

// CORRECT -- class names start with a Capital letter / 正解 -- クラス名は大文字で始める
class Ball { }

Debugging Tip / デバッグのヒント: Use console.log(ball) in the Developer Tools (F12) to inspect an object and see all of its current properties at once!

開発者ツール（F12）でconsole.log(ball)を使い、オブジェクトの現在のすべてのプロパティを 一度に確認しましょう！

16. Key Takeaways / 重要ポイント

  • OOP bundles data and behavior into self-contained objects, instead of scattering loose variables and functions.
    OOPはデータと振る舞いを自己完結型のオブジェクトにまとめる。
  • A class is a blueprint; an object is a real thing built from it with new.
    クラスは設計図、オブジェクトはnewで作られた本物。
  • The constructor() runs automatically and sets the object's starting properties.
    constructor()は自動的に実行され、初期プロパティを設定する。
  • this means "this specific object" — how an object refers to its own properties.
    thisは「この特定のオブジェクト」を意味する。
  • Methods are functions that belong to an object — call them with a dot: ball.draw().
    メソッドはオブジェクトに属する関数 — ドットで呼ぶ：ball.draw()。
  • One blueprint, many objects — new Ball(...) makes as many balls as you want.
    1つの設計図から多数のオブジェクト — 好きなだけボールを作れる。
